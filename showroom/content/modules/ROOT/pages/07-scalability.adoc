= Scalability

CertChain is designed for horizontal scalability at multiple layers — from adding new organizations to scaling individual components.

== Adding a New Organization

The App-of-Apps Helm structure makes adding a 4th (or 5th, 6th...) organization straightforward. Each org is a parameterized instance of the same `certchain-org` Helm chart.

=== What It Takes

To add a new organization called "QuantumLeap":

. *Add org config to `helm/values.yaml`*:
+
[source,yaml]
----
components:
  orgs:
    - name: quantumleap
      displayName: "QuantumLeap Labs"
      mspId: QuantumLeapMSP
      adminEmail: admin@quantumleap.demo
----

. *Create a values override file* at `helm/components/certchain-org/values-quantumleap.yaml` with org-specific settings (Keycloak realm name, display name, peer/orderer identity references).

. *Push to git* — ArgoCD detects the change and auto-generates a new `certchain-quantumleap` Application.

. *Update Fabric config* — The `configtx.yaml` needs the new org's MSP definition, and a channel configuration update transaction must be submitted.

=== What Gets Auto-Generated

The App-of-Apps template in `helm/templates/applications.yaml` iterates over the org list and generates one ArgoCD Application per org. Each org Application deploys:

* Peer + CouchDB + Orderer (blockchain nodes)
* cert-admin-api + course-manager-ui (application layer)
* Keycloak + Postgres (identity)
* RBAC, ExternalName services, enrollment jobs

== Horizontal Scaling

Fabric peers and orderers are deployed as standard Kubernetes Deployments, so scaling them is as simple as increasing the replica count. The database (CouchDB) is paired 1:1 with each peer and does *not* need independent scaling.

=== Scaling Peers

Each organization runs a single peer in this demo. In production, you can add peer replicas to increase endorsement capacity and query throughput:

[source,bash]
----
# Scale TechPulse peers to 2 replicas
oc scale deployment peer-deployment -n {central_namespace}-techpulse --replicas=2
----

Each peer replica:

* Maintains a full copy of the ledger via gossip sync
* Has its own CouchDB sidecar for state queries — no shared database needed
* Can independently endorse transactions and serve read queries
* Is load-balanced by the Kubernetes Service for incoming gRPC connections

=== Scaling Orderers

The orderer cluster uses SmartBFT consensus with 4 nodes (tolerating 1 Byzantine fault). You can add orderers to increase fault tolerance:

[cols="1,1,1"]
|===
| Orderers | Faults Tolerated | How to Add

| 4 (this demo)
| 1
| Default — one orderer per org + central

| 7
| 2
| Add 3 orderers across orgs, update `configtx.yaml` consenter set

| 10
| 3
| Add 6 orderers, requires channel config update transaction
|===

Adding orderers increases *fault tolerance* (more Byzantine faults survived), but does not increase throughput linearly — consensus round-trips grow with the number of nodes.

=== Scaling Application Services

The API and UI services are stateless and scale freely:

[source,bash]
----
# Scale verify-api for higher read throughput
oc scale deployment verify-api -n {central_namespace} --replicas=3

# Scale cert-admin-api for higher write throughput
oc scale deployment cert-admin-api -n {central_namespace}-techpulse --replicas=2
----

[cols="1,2"]
|===
| Component | Scaling Notes

| *verify-api*
| Stateless — each replica connects to the same peer. OpenShift Service load-balances.

| *cert-admin-api*
| Each replica submits transactions independently to its org's peer.

| *cert-portal / course-manager-ui*
| Static frontends — scale freely, no backend state.

| *Chaincode (CcaaS)*
| Deployed as a Kubernetes Deployment — scale replicas for concurrent endorsement execution.
|===

=== What Does NOT Need Scaling

* *CouchDB* — Paired 1:1 with each peer. Scaling peers automatically creates new CouchDB instances.
* *Fabric CA* — Only used during enrollment (startup). Not in the transaction path.
* *Keycloak / Postgres* — Identity layer operates independently from blockchain throughput.
